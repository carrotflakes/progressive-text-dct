# 実験タスク: DCTベースのプログレッシブ・テキスト圧縮（フルスクラッチ版）

## 背景とゴール

プログレッシブJPEGのように「先頭から読むほど忠実になる」テキスト圧縮を、**学習済みモデルを一切使わず**に検証する実験です。

アイデア:
- トークンを**学習可能な埋め込み**でベクトル列にし、**系列方向にDCT**をかける
- DCT係数ベクトルの**先頭K個**だけを圧縮表現とする（粗→細の順序付けはDCTに外注する）
- デコーダは**スクラッチ学習する小型Transformer LM**。切り詰めた係数列から元の文章を再構成するよう学習する
- 学習時にKをランダムにサンプリングする（Matryoshka型prefix-drop学習）ことで、任意のKで動くようにする
- **重要**: DCTと切り詰めは線形・微分可能なので、勾配は埋め込みまで流れる。つまり埋め込み空間自体が「DCT切り詰めに耐える」よう端から端まで最適化される。これがこの実験の独自ポイント

検証したい仮説:

1. **H1 (プログレッシブ性)**: Kを増やすと復元品質が単調に向上し、十分大きいKでほぼ完全復元できる
2. **H2 (グレースフルな劣化)**: Kが小さいとき、出力は「壊れた文」ではなく「大意を捉えた文」になる
3. **H3 (埋め込みの適応)**: 埋め込みを学習対象にすると、凍結ランダム埋め込みより小Kでの品質が有意に高くなる（＝埋め込みがDCTボトルネックに適応する）
4. **H4 (DCT順序の寄与)**: 先頭K係数は「ランダムK係数」や「テキスト先頭Kトークン」より意味保存に優れる

## 環境・前提

- Python 3.10+, PyTorch, tokenizers, datasets, scipy, sacrebleu, sentence-transformers, matplotlib
- GPU 1枚（bf16）、VRAM 24GB想定。足りない場合はモデル幅・バッチで調整可
- **学習済みの重みは一切使わない**。例外は評価専用ツールのみ（後述のsentence-transformersは復元品質の測定にだけ使い、学習・圧縮パイプラインには関与させない）
- トークナイザも自前で学習する: `tokenizers` ライブラリで学習コーパス上にBPE（vocab 16,384）を学習。これは統計的前処理であり「学習済みモデル」には該当しないものとする
- 乱数シード固定（42）。すべての設定は `config.yaml` に集約すること

## 手法の仕様

### エンコーダ（埋め込み層のみ・学習対象）

1. テキストをBPEでトークナイズ（長さ n、最大128）
2. 学習可能な埋め込みテーブル E ∈ R^{V×d_emb}（d_emb=512、ゼロから初期化＝通常のnormal init）でトークン列を H ∈ R^{n×d_emb} に変換。文脈化層は置かない（純粋に「埋め込み→DCT」の構成を検証する）
3. H に対し**埋め込み次元ごとに独立に系列方向のDCT-II（orthonormal正規化）**を適用する。PyTorch内で微分可能に実装すること（DCTは固定行列との行列積として書けるので、長さnごとのDCT行列をキャッシュして `H_dct = D_n @ H` とするのが簡単。orthonormal正規化は必須。Kや n でスケールが変わると学習が不安定になる）
4. 先頭K行を圧縮表現 Z ∈ R^{K×d_emb} とする
5. **勾配を切らないこと**。デコーダの損失から Z → DCT → E まで逆伝播させ、埋め込みをend-to-endで学習する

### デコーダ（スクラッチ学習のTransformer）

1. アーキテクチャ: Pre-LN Transformer、d_model=512、8層、8ヘッド、FFN 2048、RoPE、重み共有なし。パラメータ数はおよそ40M前後になる想定
2. 入力プロジェクション: 線形層 R^{d_emb→d_model} で各係数ベクトルを写像し、**学習可能な係数インデックス埋め込み**（最大K_max個）を加算する
3. 元テキストのトークン長 n を**学習可能な長さ埋め込み**1ベクトルとして条件に加える（逆DCTのスケールがnに依存するため必須）
4. 入力系列: `[長さ埋め込み] [z_1] ... [z_K] [BOS] t_1 t_2 ...`。プレフィックス部（長さ埋め込み〜z_K）は相互に全結合アテンション、テキスト部はcausalアテンション＋プレフィックスへの全アクセス、とする（prefix-LM形式）
5. t_1 以降に教師強制でCE損失
6. デコーダの入力トークン埋め込み・出力ヘッドは、エンコーダの埋め込みテーブル E とは**独立に持つ**（共有は後述のアブレーション）

### 学習

- データ: `wikitext-103-raw-v1` の train split を32〜128トークンのチャンクに分割し全量使用。validation 2000件、test 2000件を確保
- K_max = 64。サンプルごとに K を対数一様にサンプリング: `K = round(exp(uniform(log 1, log K_max)))`
- スクラッチ学習なのでLoRA版より多く回す: 100kステップ（実効バッチ64）を基準とし、validation lossの推移を見て判断。学習曲線を必ず保存
- AdamW, lr=3e-4, warmup 2000 step, cosine decay, weight decay 0.1, grad clip 1.0
- 埋め込みテーブル E の学習率は他と同じでよいが、lossが暴れる場合は E のみ半分にする選択肢を許可

### サニティチェック（Phase 1 で必ず実施）

本学習の前に2段階で確認する:

1. **可逆性テスト（学習なし）**: ランダムな H に対し DCT→K=n で全係数保持→逆DCT が数値誤差の範囲で H を復元することをユニットテストにする
2. **オーバーフィットテスト**: 100サンプル・K=K_max固定で学習し、復元トークン精度がほぼ100%に到達することを確認。到達しない場合はスケール正規化・長さ条件・アテンションマスクのバグを疑い、修正するまで先に進まないこと

## 評価

test 2000件、K ∈ {1, 2, 4, 8, 16, 32, 64} で以下を測定:

1. **トークン完全一致率**（greedy自由生成と正解の一致率）
2. **chrF / BLEU**（sacrebleu）
3. **意味類似度**: `sentence-transformers/all-MiniLM-L6-v2` による原文と復元文のコサイン類似度（評価専用。学習には不使用）
4. 3指標の **K に対する曲線**を1枚の図に（x軸 log スケール）

### ベースライン・アブレーション（同一test set・同一学習予算）

- **B1: 学習なし復号** — 先頭K係数以外をゼロ埋め→逆DCT→学習済み埋め込みテーブル E との最近傍でトークン復号。デコーダLMを使わない場合の破綻を示す（H2/H3の対照）
- **B2: 凍結ランダム埋め込み** — E をランダム初期化のまま凍結し、デコーダだけ学習。メイン手法との差分が「埋め込みのDCT適応」の効果（H3の本体）
- **B3: ランダムK係数** — 先頭ではなくランダムなK個の係数で学習・評価（係数インデックス埋め込みで位置は伝える）。低周波優先という順序自体の寄与を切り分ける（H4）
- **B4: 先頭切り詰め** — 圧縮表現として「テキスト先頭Kトークンの埋め込み」をそのまま使い同様に学習（H4）
- **A1（時間が許せば）: 埋め込み共有** — エンコーダ埋め込み E とデコーダの入力埋め込み・出力ヘッドを共有した場合の影響

### 定性評価

K ∈ {1, 4, 16, 64} について同一原文5本の復元結果を並べた表を作り、「小Kで大意が残るか・文法が崩れるか」を目視確認できる形にする。

### 追加の観察（H3の深掘り、安価なので必ず実施）

学習後の埋め込みテーブル E について、B2のランダム埋め込みと比較して以下を可視化する:
- 実テキストの埋め込み列 H のDCT係数エネルギー分布（周波数別の平均パワー）。学習後に低周波へエネルギーが寄っていれば「埋め込みがDCT向きに変形した」直接証拠になる
- 高頻度語・機能語と内容語で埋め込みノルムや分布に差が出ているか

## 成果物

```
progressive-text-dct-scratch/
  config.yaml
  src/            # tokenizer.py, data.py, dct.py, model.py, train.py, evaluate.py
  scripts/        # run_sanity.sh, run_train.sh, run_eval.sh
  results/        # metrics.csv, curves.png, spectrum.png, samples.md
  report.md
```

`report.md` には以下を含める:
- 実験設定の要約（パラメータ数、データ量、学習時間、ハードウェア）
- 品質 vs K の曲線（メイン + B1〜B4）
- DCT係数エネルギー分布の比較図（メイン vs B2）
- 定性サンプル表
- 仮説H1〜H4それぞれの判定と根拠
- 設計判断ログ（独断で決めた箇所すべてに理由を記録）
- 失敗・予想外の挙動も隠さず記録すること

## 進め方と制約

- Phase 0: 環境構築・トークナイザ学習・データ準備 → Phase 1: パイプライン実装＋サニティチェック2種 → Phase 2: 本学習（メイン + B2 + B3 + B4） → Phase 3: 評価・スペクトル分析・レポート
- 学習ジョブはtmuxセッション内で起動し、定期的にログを確認して進捗を報告すること
- チェックポイントは15分ごとに保存し、中断から再開可能にすること
- 各Phase完了時に中間結果を `results/` に保存すること
- 学習1本あたり数時間以内を想定。超えそうな場合はステップ数・モデル幅を比例縮小し、縮小したことをreportに明記する（4本の学習がすべて同一予算であることが比較の前提なので、縮小はすべての設定に一律に適用すること）
- 評価指標は既存ライブラリ（sacrebleu, sentence-transformers）を使い自作しないこと
