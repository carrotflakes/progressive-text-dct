# 実験タスク: DCTベースのプログレッシブ・テキスト圧縮

## 背景とゴール

プログレッシブJPEGのように「先頭から読むほど忠実になる」テキスト圧縮を検証する実験です。

アイデア:
- 文章をLMの埋め込み（または浅い層の隠れ状態）列に変換し、**系列方向にDCT**をかける
- DCT係数ベクトルの**先頭K個**だけを圧縮表現として保持する（粗→細の順序付けをDCTに外注する）
- **学習するのはデコーダ側だけ**: 切り詰めた係数列から元の文章を生成するようLMをLoRAでファインチューニングする
- 学習時にKをランダムにサンプリングする（Matryoshka型prefix-drop学習）ことで、任意のKで動くデコーダを作る

検証したい仮説:

1. **H1 (プログレッシブ性)**: Kを増やすと復元品質が単調に向上し、十分大きいKでほぼ完全復元できる
2. **H2 (グレースフルな劣化)**: Kが小さいとき、出力は「壊れた文」ではなく「大意を捉えた文」になる
3. **H3 (学習デコーダの必要性)**: 逆DCT＋最近傍トークン復号（学習なし）は小さいKで破綻する
4. **H4 (DCT順序の意味)**: 先頭K係数は「テキスト先頭Kトークンだけ見せる」ベースラインより意味保存に優れる

## 環境・前提

- Python 3.10+, PyTorch, transformers, peft, datasets, scipy, sentence-transformers, matplotlib
- GPU 1枚（bf16）。VRAM 24GB想定。足りない場合はバッチサイズ・モデルサイズで調整可
- ベースLM: `Qwen/Qwen2.5-0.5B`（第一候補）。ロードできない場合は `gpt2`(124M) にフォールバック
- 乱数シード固定（42）。すべての設定は `config.yaml` に集約すること

## 手法の仕様

### エンコーダ（学習なし・凍結）

1. テキストをトークナイズし、トークン列（長さ n、n は最大128）を得る
2. ベースLMで隠れ状態を取得する。`encoder_layer` を設定可能にする:
   - `encoder_layer = 0`: 埋め込み層の出力（生埋め込み）
   - `encoder_layer = L`: 第L層通過後の隠れ状態（デフォルト L=4 程度）
3. 隠れ状態行列 H ∈ R^{n×d} に対し、**埋め込み次元ごとに独立に系列方向のDCT**を適用する
   - `scipy.fft.dct(H, type=2, norm='ortho', axis=0)` 相当。PyTorch内で完結させたい場合は自前実装か `torch-dct` を使用（orthonormal正規化を必ず使うこと。Kによってスケールが変わると学習が不安定になる）
4. 先頭K行（低周波側のK個の係数ベクトル）を圧縮表現 Z ∈ R^{K×d} とする

### デコーダ（学習対象）

1. 入力プロジェクション: 線形層 W ∈ R^{d×d_model}（学習可能）で各係数ベクトルをLMの埋め込み空間へ写像する
2. 各係数位置に**学習可能な係数インデックス埋め込み**（最大K_max個）を加算する。LM本来の位置埋め込み/RoPEとは別に持つこと
3. 元テキストのトークン長 n を条件として与える（学習可能な長さ埋め込みを1ベクトル追加する方式でよい。逆DCTのスケールはnに依存するため必須）
4. デコーダLMへの入力系列: `[長さ埋め込み] [z_1] ... [z_K] [BOS] t_1 t_2 ...` とし、t_1 以降を自己回帰で生成・教師強制でCE損失を計算する
5. デコーダLMはLoRA（r=16, alpha=32, 全attention/MLP線形層対象）で学習。入力プロジェクション・係数インデックス埋め込み・長さ埋め込みはフル学習

### 学習

- データ: `wikitext-103-raw-v1` の train split。32〜128トークンのチャンクに分割し、約20万チャンクを使用。validation 2000件、test 2000件を確保
- K_max = 64。学習時、各サンプルごとに K を対数一様にサンプリング: `K = round(exp(uniform(log 1, log K_max)))`（小さいKを十分踏ませるため。一様だと小Kの学習が不足する）
- ステップ数: まず2万ステップ（バッチ32相当）。validation lossが下がり続けていれば延長可。学習曲線を保存すること
- AdamW, lr=1e-4（プロジェクション層）/ 2e-4（LoRA）, warmup 500 step, cosine decay

### サニティチェック（Phase 1 で必ず実施）

本学習の前に、100サンプルだけで K=K_max 固定でオーバーフィットさせ、**復元トークン精度がほぼ100%に到達する**ことを確認する。到達しない場合はスケール正規化・長さ条件・マスク処理のバグを疑い、修正するまで先に進まないこと。

## 評価

test 2000件に対し、K ∈ {1, 2, 4, 8, 16, 32, 64} の各点で以下を測定する:

1. **トークン完全一致率**（自由生成、greedy decoding、生成と正解のトークン列一致率）
2. **chrF および BLEU**（sacrebleu）
3. **意味類似度**: `sentence-transformers/all-MiniLM-L6-v2` で原文と復元文の埋め込みコサイン類似度
4. 上記3指標の **K に対する曲線**を1枚の図にまとめる（x軸は log スケール）

### ベースライン（同一test setで比較）

- **B1: 学習なし復号** — 係数を先頭K個でゼロ埋め→逆DCT→各位置で語彙埋め込みとの最近傍トークンを取る。H3の検証用（encoder_layer=0 の設定でのみ意味を持つ点に注意し、メイン設定と別に encoder_layer=0 の学習済みモデルも1本用意して条件を揃えること）
- **B2: 先頭切り詰め** — 圧縮表現として「テキストの先頭Kトークンの隠れ状態」をそのまま使い、同じ手順で学習したデコーダ。H4の検証用
- **B3: ランダムK係数** — 先頭ではなくランダムなK個の係数を使って学習・評価（係数インデックス埋め込みで位置は伝える）。「低周波から順に」という順序自体の寄与を切り分ける

### 定性評価

K ∈ {1, 4, 16, 64} について、同一原文5本の復元結果を並べた表を作る。「Kが小さいとき大意は残るか、文法は崩れるか」を目視確認できる形にすること。

## 成果物

リポジトリ構成:

```
progressive-text-dct/
  config.yaml
  src/            # data.py, model.py, train.py, evaluate.py
  scripts/        # run_sanity.sh, run_train.sh, run_eval.sh
  results/        # metrics.csv, curves.png, samples.md
  report.md
```

`report.md` には以下を含めること:
- 実験設定の要約（モデル、データ量、学習時間、ハードウェア）
- 品質 vs K の曲線（メイン手法 + B1/B2/B3）
- 定性サンプル表
- 仮説H1〜H4それぞれに対する判定と根拠
- encoder_layer=0 と encoder_layer=4 の比較（時間が許せば。許さなければ encoder_layer=4 のみでよく、その旨を明記）
- 失敗・予想外の挙動も隠さず記録すること

## 進め方と制約

- Phase 0: 環境構築・データ準備 → Phase 1: パイプライン実装＋サニティチェック → Phase 2: 本学習（メイン + B2 + B3） → Phase 3: 評価・レポート
- 各Phaseの完了時に中間結果を `results/` にコミットすること
- 学習1本あたりの想定時間は数時間以内。超えそうな場合はデータ量・ステップ数を比例縮小し、縮小したことをreportに明記する
- 不確実な設計判断（正規化方法、長さ条件の与え方など）は、独断で進めてよいが必ず `report.md` の「設計判断ログ」節に理由とともに記録すること
- 評価指標の実装を自作せず、既存ライブラリ（sacrebleu, sentence-transformers）を使うこと
